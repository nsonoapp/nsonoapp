INDEXS FIRESTORE — NSOSO / NSONO
Référence unique pour la console Firebase et firestore.indexes.json
Format : chaque champ indique explicitement ASC ou DESC


---

✅ RÈGLE PRO ERP FIREBASE

Quand un champ sert souvent de filtre (where)

ET

que les résultats sont souvent triés par date (orderBy)

➡️ alors : filtre ASC, createdAt DESC

devient presque le standard ERP.

Exemples réalistes :

sellerId ASC, createdAt DESC
status ASC, createdAt DESC
category ASC, createdAt DESC
productId ASC, createdAt DESC

Exception FIFO stock : productId ASC, createdAt ASC

✅ PRIORITÉ DÉPLOIEMENT (bloquant → utile)

  0. role ASC, approvalStatus ASC                              → users (admin entités)
  1. sellerId ASC, createdAt DESC                              → sales (stats + vendus)
  2. productId ASC, createdAt DESC                             → sale_items (vendus)
  3. payment_status ASC, createdAt DESC                        → sales (vendus)
  4. status ASC, createdAt DESC                                → sales (vendus)
  5. sellerId ASC, payment_status ASC, status ASC, createdAt DESC
  6. amount_remaining DESC                                       → debts (top dettes stats)
  7. productId ASC, createdAt ASC                              → stock_movements (FIFO)
  8. status ASC, DueDate ASC                                     → debts (échéances)
  9. category ASC, createdAt DESC                                → expenses
 10. entityId ASC, createdAt DESC                               → logs
 11. le reste (losses, stock_movements type/reason, futurs)

✅ CONCLUSION PRO NSOSO

Dans cet ERP, les filtres finissent presque toujours avec createdAt DESC.

Règle stricte :
  ✅ index composite seulement si le champ est dans un where() réel
  ❌ pas d’index pour tous les champs du schéma

Note technique :
  Les index mono-champ (createdAt DESC seul, amount_remaining DESC seul)
  sont gérés automatiquement par Firestore — pas besoin dans indexes.json.
